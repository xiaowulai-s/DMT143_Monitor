# DMT143 连接问题修复与测试指南

## 📋 问题总结

**主要问题**: 连接设备后不显示数据

**根本原因**: 
1. `send_command` 方法在发送命令前会调用 `_send_stop()`,这会导致停止设备的连续输出
2. 连接流程中等待时间不足,设备可能来不及响应
3. 缺少详细的原始数据日志,难以诊断问题

## ⚠️ 常见问题排查

### 问题1: 无任何响应

**可能原因**:
- 串口选择错误
- 波特率不匹配
- 设备未通电

**解决方案**:
```bash
# 1. 确认设备管理器中的串口号
# 2. 尝试不同的波特率 (9600, 19200, 38400)
# 3. 检查设备电源和指示灯
```

### 问题2: 设备信息查询失败

**可能原因**:
- 设备在输出模式,无法响应命令

**解决方案**:
```python
# debug_serial.py 会先发送停止命令
# 如果仍然失败,手动操作:
# 1. 断开程序
# 2. 重新连接设备电源
# 3. 快速运行 debug_serial.py
```

### 问题3: 格式设置返回 None

**可能原因**:
- 设备响应时间不足
- 设备固件版本不支持

**解决方案**:
```python
# 已在修复中增加等待时间
# 如果仍然返回 None,可以忽略,直接使用默认格式
```

### 问题4: 有原始数据但解析失败

**可能原因**:
- 数据格式与预期不符

**解决方案**:
```python
# 1. 查看原始数据格式
[DMT143] <<< 原始数据(45字节): "Tdf= 6.26 'C ..."

# 2. 如果格式不同,需要修改 parse_data 方法
# 将原始数据格式反馈给开发者
```

### 问题5: 数据显示不更新

**可能原因**:
- UI 刷新问题
- 数据未传递到界面

**解决方案**:
```python
# 检查是否看到日志:
[DMT143] >>> 解析成功: {'dewpoint': 6.26, ...}

# 如果有,但界面不更新:
# 1. 检查 refresh_timer 是否启动
# 2. 检查 on_data_received 是否被调用
# 3. 重启程序
```
